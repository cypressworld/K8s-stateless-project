📚 Lesson Notes: Chapter 11 – Deploying Microservices
1. Introduction: Deploying microservices splits into two related but distinct realms:

Production Deployments: The core focus is system stability, predictable builds, and deployment predictability. Most importantly, the operations team must have the ability to roll back bad deployments instantly to minimize downtime.

Development Deployments: The core focus is developer velocity. Developers need a fast turnaround time and isolated environments so they don't clutter shared source control or interfere with each other's code.

2. The Bridge: CI/CD Systems
To bridge these two worlds, modern cloud-native architectures rely on a continuous integration and continuous deployment (CI/CD) pipeline. This system automates everything from pulling code from source control, baking it into secure images registries, and pushing temporary versions to development environments versus stable versions to production.

3. Core Kubernetes Deployment Strategies
When launching or upgrading microservices in production, Kubernetes offers different strategies to update applications without crashing the system:

Rolling Updates (Default): Gradually replaces old instances (Pods) with new versions one by one. Zero downtime, but two different versions of your code will run concurrently for a brief window.

Blue-Green Deployment: You deploy the brand-new version (Green) alongside the old version (Blue) in full capacity. Once the Green version passes QA checks, you flip a router switch (Kubernetes Service) to send 100% of the traffic to Green. If something breaks, you flip it back instantly.

Canary Deployment: You route a tiny fraction of real user traffic (e.g., 5%) to the new version. If no errors are triggered, you gradually scale it up to 100%.

  name: microservice-blue
  labels:
    app: my-microservice
    version: blue

  replicas: 2
  selector:
    matchLabels:
      app: my-microservice
      version: blue

    metadata:
      labels:
        app: my-microservice
        version: blue
    spec:
      containers:
      - name: web-app
        image: nginx:alpine
        ports:
        - containerPort: 80
        lifecycle:
          postStart:
            exec:

Step 2: Create the Routing Service
Create a file named production-routing-service.yaml. Notice that the selector specifically targets version: blue.

YAML
apiVersion: v1
kind: Service

  type: NodePort
  ports:
  - port: 80
    targetPort: 80
    nodePort: 30080

    app: my-microservice

Deploying Stateless Applications
1. The Core Lifecycle of a Kubernetes Deployment
A Deployment controller is the standard declarative configuration tool used to manage stateless applications. It sits above a ReplicaSet, which monitors pod counts, ensuring the exact numbers of healthy replicas are always alive inside the nodes.

                  ┌─────────────────┐
                  │   Deployment    │
                  └────────┬────────┘
                           │ Manages
                  ┌────────▼────────┐
                  │   ReplicaSet    │
                  └────────┬────────┘
                           │ Monitors
         ┌─────────────────┼─────────────────┐
         │                 │                 │
    ┌────▼────┐       ┌────▼────┐       ┌────▼────┐
    │  Pod 1  │       │  Pod 2  │       │  Pod 3  │
    └─────────┘       └─────────┘       └─────────┘
Imperative Dry-Run Creation: You can quickly output a boilerplate skeleton configuration structure using the --dry-run=client -o yaml flags to verify layout properties before pushing it to the api-server.

The Rollout Track: Running kubectl rollout status allows programmatic scripts or engineers to block execution until all node replicas report a READY state.

2. Service Discovery and Network Endpoints
A Service acts as an internal or public load balancer pointing traffic toward dynamic, volatile Pod IPs.

The Selector Nexus: Services hook into Pods using standard metadata key-value matches (e.g., app=nginx,environment=test).

The Endpoint List: Running kubectl describe service extracts the exact real-time IP list (Endpoints) currently mapped to the backend array. If a Pod fails its health check, its IP is scrubbed from this matching registry instantly.

3. Traffic Verification via Readiness Probes
A Readiness Probe determines if a container is legally allowed to accept incoming network requests.

The Operational Rule: If a container is technically running but its readiness probe evaluates to a failure state (e.g., HTTP 404), Kubernetes drops its connectivity indicator (READY 0/1).

The Consequence: The Service removes the pod's IP from the Endpoints pool. Clients hitting the app gateway will not be routed to unready containers, avoiding server errors.

4. Advanced Update Rollouts and Fail-Safes
Rolling Update Process: Gradually replaces old container image rings with upgraded software version labels. By controlling maxSurge and maxUnavailable boundaries, you guarantee continuous operations.

The History Track and Rollbacks: The cluster logs past modifications under kubectl rollout history. If a bug skips past testing blocks, running kubectl rollout undo reverses the deployment structure to an older revision signature in seconds.

Class Summary:
Stateful: Stores context locally. Hard to scale, hard to update, fragile if it crashes (e.g., Databases like MySQL, PostgreSQL, Redis).

Stateless: Stores no context locally. Offloads data to an external database. Easy to scale, easy to replace, perfectly optimized for cloud platforms like Kubernetes (e.g., Frontend apps, UI servers, Business logic REST APIs).

🧠 Deep-Dive: What These Files Actually Do
To help your students understand what they are interacting with when they import this repository, explain the files like this:

1. The Custom Image Blueprint Files (Containerfile & index.html)
Containerfile: (This is identical to a Dockerfile). It contains the raw code instructions used to build a custom container image. It tells the builder tool (like Podman or Docker) to grab standard Nginx and inject the custom index.html file into it.

index.html: The plain-text webpage that displays when you visit the server. Your appendix notes show this file being compiled via podman build -t nginx-custom:1.0 ..

2. The Deployment Architecture Blueprints
nginx-deployment.yaml: This is the absolute starting point. It creates your basic stateless application pool (your 3 identical replica pods running Nginx) without any strict health probes.

nginx-deployment-readinessprobe.yaml: This is the exact file you used in your practical simulation. It contains the readinessProbe rule looking for the /ready path token. This is where students learn how to simulate a service failure and repair it using the touch command.

nginx-deployment-rollingupdate.yaml: This file is configured to test upgrades. Students apply this file first with an old version (like nginx:1.18), change the text string inside to nginx:1.19, re-apply it, and run kubectl rollout status to watch the cluster drop old containers and spin up new ones one-by-one.

3. The Network Gateway Blueprints
nginx-service.yaml: Configures a standard cloud-native LoadBalancer. It gives your pod replicas a single unified IP address and distributes customer traffic evenly among them.

nginx-service-nodeport.yaml: An alternative networking style. Instead of a cloud load balancer, a NodePort service punches a dedicated hole through the actual physical host computer’s network card (allocating a port like 32563) so anyone on the local office network can type the computer's IP address and access the website.

***Error
Plaintext
error: unknown flag: --sleep
The kubectl run command does not natively understand a --sleep flag. Because that flag threw an error, Kubernetes aborted the command and never actually created the cluster-tester pod. That is why every command after it kept screaming Error from server (NotFound).

Let's fix this instantly with the correct Kubernetes syntax. Run these three commands line-by-line in your terminal to create the tester pod properly and fire your live traffic request:

1. Launch the tester pod correctly
Instead of using a separate flag, we pass the sleep command directly to the container's arguments using -- like this:

Bash
kubectl run cluster-tester --image=curlimages/curl -- sh -c "sleep 3600"
2. Wait 5 seconds for it to boot up
Bash
kubectl wait --for=condition=Ready pod/cluster-tester --timeout=15s
(You should see a message saying pod/cluster-tester condition met)

3. Curl your service endpoint again (The Traffic Test!)
Bash
kubectl exec -it cluster-tester -- curl http://nginx-service-example.default.svc.cluster.local
Once you run that final command, the cluster-tester pod will fetch the data from your load balancer, and you will see the raw HTML of the Nginx welcome page stream directly onto your screen!
